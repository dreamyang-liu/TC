贪心算法

每步做局部最优选择，且能证明通向全局最优

难点不在代码，在判断能不能贪。贪心的代码通常很短，但正确性证明是灵魂。
练习进度： 已完成 0/10

一、两种证明方式

1. Exchange Argument（交换论证）

思路：假设存在一个最优解 OPT 和贪心解 G 不同，找到第一个不同的地方，把 OPT 中的选择换成 G 的选择，证明结果不会变差。既然 OPT 已经是最优，换了也不差 → 矛盾或证明 G 也是最优。

适用场景 排序类贪心（如按截止时间排序的调度问题）

2. Greedy Stays Ahead（贪心始终领先）

思路：用数学归纳法证明贪心在每一步都不比最优解差。既然每步都领先或持平，最终结果自然最优。

适用场景 每步有可量化指标的问题（如区间调度选的区间数）

二、5个常见套路

  1. 按某 key 排序后扫一遍 — 区间问题按右端点排序是经典
  2. 双堆/优先队列 — 动态维护最大/最小值（如 IPO 问题）
  3. 抵消/Cancel — Boyer-Moore 投票算法，多数元素
  4. 反悔贪心 — 先贪心选，遇到更好的反悔替换（课程表III）
  5. 双指针贪心 — 两端或快慢指针（分配饼干问题）

四、反悔贪心模板（课程表III）

口诀：先贪婪装，装不下就扔最贵的
def schedule_courses(courses):
    courses.sort(key=lambda c: c[1])  # 按 deadline 排序
    total = 0
    heap = []
    for duration, deadline in courses:
        total += duration
        heapq.heappush(heap, -duration)  # 最大堆
        if total > deadline:
            total += heapq.heappop(heap)  # 弹出最大的（负数所以是加）
    return len(heap)

核心思想：遍历时无脑加入，如果超时了就反悔——把之前选过的最长课程踢掉，总时间一定会变小（因为踢掉的 ≥ 当前的或能让总时间回到 deadline 内）。

五、贪心 vs DP

简单判断：不确定能贪就回退用 DP。

特征 贪心 DP
选择 当前最优，不回头 尝试所有子问题
正确性 需要证明 天然正确（穷举）
复杂度 通常 O(n log n) 通常 O(n^2) 或更高
典型信号 排序后扫描、优先队列 最优子结构 + 重叠子问题

六、思考题

"交换论证"和"贪心始终领先"两种证明方式各适合什么场景？

交换论证适合"排列/排序型"问题——你需要证明某种排序方式是最优的（如按 deadline、按 ratio 排序）。贪心始终领先适合"选择型"问题——每步选一个元素，需要证明你选的集合在每一步都不比最优解差（如区间调度）。

为什么反悔贪心要用最大堆？如果用最小堆会怎样？

反悔时要踢掉"代价最大"的已选元素——踢掉最长的课程能腾出最多时间。如果用最小堆踢掉最短的，腾出的时间最少，可能还是超时，而且损失了一个"便宜"的选择。最大堆保证每次反悔的收益最大化。

怎么快速判断一道题能不能用贪心？

三步法：1) 能否找到一个排序/选择策略使得每步的局部最优不影响后续？ 2) 能否构造反例推翻贪心？如果试了几个反例都推不翻，大概率能贪。3) 题目是否有"最小/最大数量"且输入可排序？这些都是贪心的信号。实在不确定就写 DP。
