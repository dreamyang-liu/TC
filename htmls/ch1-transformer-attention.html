1.1 Self-Attention 机制

Self-Attention 是 Transformer 的核心计算单元。它允许序列中的每个 token "关注" 其他所有 token，从而捕获长距离依赖关系。其核心公式为：

Attention(Q, K, V) = softmax(Q KT / √dk) V

其中：

  • Q (Query)：查询矩阵，表示 "我在找什么"
  • K (Key)：键矩阵，表示 "我有什么信息"
  • V (Value)：值矩阵，表示 "我能提供什么内容"
  • dk：Key 向量的维度，用于缩放防止梯度消失
为什么需要缩放？ 当 dk 较大时，Q 和 K 的点积结果方差会很大（约为 dk），导致 softmax 输出趋近于 one-hot 分布，梯度接近于零。除以 √dk 可以将方差归一化为 1。

代码实现：Scaled Dot-Product Attention

import torch
import torch.nn.functional as F
import math

def scaled_dot_product_attention(Q, K, V, mask=None):
    """
    手动实现 Scaled Dot-Product Attention

    Args:
        Q: Query tensor, shape (batch, seq_len_q, d_k)
        K: Key tensor, shape (batch, seq_len_k, d_k)
        V: Value tensor, shape (batch, seq_len_k, d_v)
        mask: 可选的 attention mask

    Returns:
        output: shape (batch, seq_len_q, d_v)
        attention_weights: shape (batch, seq_len_q, seq_len_k)
    """
    d_k = Q.size(-1)

    # Step 1: 计算 Q @ K^T，得到 attention scores
    scores = torch.matmul(Q, K.transpose(-2, -1))  # (batch, seq_q, seq_k)

    # Step 2: 缩放
    scores = scores / math.sqrt(d_k)

    # Step 3: 应用 mask（用于 causal attention）
    if mask is not None:
        scores = scores.masked_fill(mask == 0, float('-inf'))

    # Step 4: Softmax 归一化
    attention_weights = F.softmax(scores, dim=-1)

    # Step 5: 加权求和 V
    output = torch.matmul(attention_weights, V)  # (batch, seq_q, d_v)

    return output, attention_weights


# 示例用法
batch_size, seq_len, d_model = 2, 8, 64
Q = torch.randn(batch_size, seq_len, d_model)
K = torch.randn(batch_size, seq_len, d_model)
V = torch.randn(batch_size, seq_len, d_model)

# Causal mask: 下三角矩阵，防止看到未来的 token
causal_mask = torch.tril(torch.ones(seq_len, seq_len))

output, weights = scaled_dot_product_attention(Q, K, V, mask=causal_mask)
print(f"Output shape: {output.shape}")       # (2, 8, 64)
print(f"Weights shape: {weights.shape}")    # (2, 8, 8)
print(f"Weights sum: {weights.sum(dim=-1)}") # 每行和为 1

1.2 Multi-Head Attention

Multi-Head Attention (MHA) 将输入投影到多个不同的子空间中，每个 "head" 独立计算 attention，最后将结果拼接并投影回原始维度。

MultiHead(Q, K, V) = Concat(head1, ..., headh) WO
where headi = Attention(QWiQ, KWiK, VWiV)

为什么要拆分多个 Head？

  • 多样性：不同的 head 可以学习关注不同类型的关系（语法、语义、位置等）
  • 计算效率：每个 head 的维度为 dk = dmodel / h，总计算量与单个大 head 相同
  • 表达能力：多个子空间的注意力模式比单一空间更丰富
典型配置：GPT-3 使用 96 个 head，dmodel=12288，每个 head 维度为 128。LLaMA-2-70B 使用 64 个 head（加上 8 个 KV head 用于 GQA）。

1.3 Flash Attention

Flash Attention 是一种 IO-aware 的精确 attention 算法。它的核心动机：GPU 的计算速度远快于内存带宽。标准 Attention 的瓶颈不是计算 FLOPS，而是 HBM（高带宽内存）的读写次数。

问题根源：标准 Attention 的 HBM 访问

标准 Attention 的计算步骤，每一步都需要 HBM 读/写。我们以 单个 head 为例，追踪每个矩阵的 shape 和对应的 HBM 访问量：

# 设 N = 序列长度, d = head 维度 (如 N=4096, d=128)
# Q: [N, d]  K: [N, d]  V: [N, d]  均存储在 HBM 中
# 每个元素占 2 bytes (FP16)

# ═══════ Step 1: 计算 Attention Score 矩阵 ═══════
# 从 HBM 读取 Q[N,d] 和 K[N,d]
#   读取量 = N×d + N×d = 2Nd 个元素
S = Q @ K.T
# S 的 shape: [N, d] × [d, N] = [N, N]
# 将 S[N,N] 写回 HBM
#   写入量 = N×N = N² 个元素  ← 这是灾难性的!

# ═══════ Step 2: Softmax ═══════
# 从 HBM 读取 S[N,N]
#   读取量 = N² 个元素
P = softmax(S / sqrt(d))
# P 的 shape: [N, N] (每行 sum=1)
# 将 P[N,N] 写回 HBM
#   写入量 = N² 个元素

# ═══════ Step 3: 加权求和 Value ═══════
# 从 HBM 读取 P[N,N] 和 V[N,d]
#   读取量 = N² + Nd 个元素
O = P @ V
# O 的 shape: [N, N] × [N, d] = [N, d]
# 将 O[N,d] 写回 HBM
#   写入量 = Nd 个元素

# ═══════ 总 HBM 访问量汇总 ═══════
# 读: 2Nd + N² + N² + Nd = 3Nd + 2N²
# 写: N² + N² + Nd       = 2N² + Nd
# 合计: 4N² + 4Nd = Θ(N² + Nd)
#
# 代入 N=4096, d=128:
#   N² = 16,777,216  |  Nd = 524,288
#   总计 ≈ 4×16.7M + 4×0.5M ≈ 69M 个元素 ≈ 132 MB
#
# 额外 HBM 内存 (存储中间结果):
#   S[N,N] + P[N,N] = 2×N² = 2×4096² = 32M 个元素 = 64 MB!
关键瓶颈：中间矩阵 S 和 P 都是 [N, N] 大小。当 N=4096, d=128 时：
• S[4096×4096] 占用 4096²×2 = 32MB
• Q, K, V 各只占 4096×128×2 = 1MB
• 中间结果比输入/输出大 32 倍！
我们为了存一个临时中间矩阵，付出了 O(N²) 的额外内存和大量的 HBM 读写。

Flash Attention 的解法：分块 + 在线 Softmax

核心思想：永远不把 N×N 矩阵写入 HBM。将 Q, K, V 切成小块，在 Shared Memory（每个 SM 上的可编程 SRAM，A100 上最大 ~192KB/SM）中完成所有计算。注意这不是 L2 Cache（40MB，硬件管理），而是程序员可直接控制的片上存储。

分块策略

设单个 SM 的 Shared Memory 容量为 M 个元素（如 A100 上 M ≈ 100K FP16 elements ≈ 192KB）。我们需要在 Shared Memory 中同时容纳：

  • K 的一个块：Bc × d 个元素
  • V 的一个块：Bc × d 个元素
  • Q 的一个块：Br × d 个元素
  • 输出 O 的一个块：Br × d 个元素
  • 中间 score 矩阵：Br × Bc 个元素
SRAM 约束: 2 × Bc × d + 2 × Br × d + Br × Bc ≤ M

最优 block size: Bc = ⌈M / (4d)⌉, Br = min(⌈M / (4d)⌉, d)

IO 复杂度推导 (核心！)

标准 Attention 的 HBM 访问量：

读 Q: Nd  |  读 K: Nd  |  写 S: N²  |  读 S: N²  |  写 P: N²  |  读 P: N²  |  读 V: Nd  |  写 O: Nd

总计 = Θ(Nd + N²) 次 HBM 访问
额外内存 = O(N²)（存储 S 和 P 矩阵）

Flash Attention 的 HBM 访问量推导：

外循环遍历 K, V 的块数 = N / Bc 次。内循环遍历 Q 的块数 = N / Br 次。

每次外循环：加载 Kj (Bc×d) + Vj (Bc×d) = 2 × Bc × d
每次内循环：加载 Qi (Br×d) + 加载/写回 Oi (Br×d) = 2 × Br × d

总 HBM 访问 = (N/Bc) × [2Bcd + (N/Br) × 2Brd]
               = (N/Bc) × [2Bcd + 2Nd]
               = 2Nd + 2N²d/Bc

代入 Bc = Θ(M/d)：
               = 2Nd + 2N²d/(M/d) = 2Nd + 2N²d²/M

∴ Flash Attention HBM 访问 = O(N²d²/M)
  额外内存 = O(N)（只需存储每行的 running max 和 running sum）
为什么这么好？ 当 M > d²（典型: M=100K, d²=16K）时，N²d²/M < N²。
例：N=4096, d=128, M=100K →
标准: 4096² + 4096×128 ≈ 17.3M 次 HBM 访问
Flash: 4096²×128²/100K ≈ 2.75M 次 HBM 访问 (节省 84%!)

数学原理：Softmax 的递推分解

设我们正在处理 Q 的第 i 行，已经遍历了前 j-1 个 KV 块。

先回顾标准 softmax attention 的最终目标：

# 标准 softmax attention (第 i 行):
output_i = Σ_k  softmax(s_k) × v_k
         = Σ_k  [exp(s_k) / Σ_j exp(s_j)] × v_k
         = (Σ_k exp(s_k) × v_k) / (Σ_k exp(s_k))
         = numerator / denominator

所以 attention 输出本质上就是 分子 / 分母。Flash Attention 的在线算法就是增量式地累积分子和分母，同时用一个 running max 来保持数值稳定：

# ═══ 维护三个统计量（处理完前 j-1 个 KV 块后）═══

m = 目前见过的所有 score 的 max               # 标量, 用于数值稳定
l = Σ exp(s_k - m)                            # 标量, rescaled 分母
O = Σ exp(s_k - m) × v_k                     # [1×d], rescaled 分子

# 注意关系: O / l = Σ softmax(s_k) × v_k = 最终 attention 输出!
# 所以 O 就是 l 的 "向量版本"，每个 v_k 的贡献按相同 weight 加权

# ═══ 处理第 j 个 KV 块 ═══
# 计算新 score: S_j = q_i × K_j^T  (在 SRAM 中)

m_new = max(m, max(S_j))                      # 可能发现更大的值
α     = exp(m - m_new)                       # correction factor (≤1)

# 更新分母: 旧的 exp 都要乘 α 修正, 再加新块的贡献
l_new = α × l  +  Σ exp(S_j - m_new)

# 更新分子: 同样 rescale 旧值, 再加新块的 exp(score) × value
O_new = α × O  +  exp(S_j - m_new) × V_j

# l 和 O 的更新公式结构完全相同!
# l 累加的是 exp(score)        → 标量
# O 累加的是 exp(score) × v_k  → 向量 (每个 score 乘对应的 value)

# ═══ 遍历完所有块后 ═══
output_i = O / l    # 分子 / 分母 = softmax attention 输出 ✓

直觉理解：

  • O 和 l 的关系 — l 累加 exp(score)，O 累加 exp(score)×v。最后 O/l 就是 softmax 加权平均。它们的更新方式完全对称
  • 为什么需要 α = exp(m_old - m_new)? — 遇到更大 score 后，之前所有 exp(s - m_old) 应该变成 exp(s - m_new)。由 exp(s - m_new) = exp(s - m_old) × exp(m_old - m_new)，所以旧值全部乘 α 即可
  • 如果新块没有更大 max? — m_new = m_old → α = 1 → 旧值不动，只加新项
  • 数学等价性 — 不是近似！遍历完所有块后 O/l 严格等于 softmax(QK^T)V

完整伪代码

# Flash Attention Forward Pass 完整算法
# 输入: Q, K, V ∈ R^{N×d}，存储在 HBM 中
# 输出: O ∈ R^{N×d}，写入 HBM
# SRAM 容量: M 个元素

# Step 1: 确定 block sizes
B_c = M // (4 * d)  # KV block 行数
B_r = min(M // (4 * d), d)  # Q block 行数
T_c = ceil(N / B_c)  # K,V 的块数
T_r = ceil(N / B_r)  # Q 的块数

# Step 2: 初始化输出和统计量（在 HBM 中）
O = zeros(N, d)
l = zeros(N)    # softmax denominator
m = full(N, -inf)  # running max

# Step 3: 外循环 - 遍历 K, V 块
for j in range(T_c):
    # 从 HBM 加载一块 K, V 到 SRAM
    K_j = K[j*B_c : (j+1)*B_c]  # [B_c, d] → HBM 读取
    V_j = V[j*B_c : (j+1)*B_c]  # [B_c, d] → HBM 读取

    # 内循环 - 遍历 Q 块
    for i in range(T_r):
        # 从 HBM 加载一块 Q, O, l, m 到 SRAM
        Q_i = Q[i*B_r : (i+1)*B_r]  # [B_r, d]
        O_i = O[i*B_r : (i+1)*B_r]  # [B_r, d]
        l_i = l[i*B_r : (i+1)*B_r]  # [B_r]
        m_i = m[i*B_r : (i+1)*B_r]  # [B_r]

        # === 以下全部在 SRAM 中完成 ===

        # 计算 attention scores (这是唯一的矩阵乘)
        S_ij = Q_i @ K_j.T  # [B_r, B_c] ← 在 SRAM 中!

        # 在线 softmax 更新
        m_new = max(m_i, S_ij.max(dim=-1))
        P_ij = exp(S_ij - m_new[:, None])
        l_new = exp(m_i - m_new) * l_i + P_ij.sum(dim=-1)

        # 更新输出 (rescale 旧结果 + 加入新贡献)
        O_i = (l_i / l_new)[:, None] * exp(m_i - m_new)[:, None] * O_i \
              + (1 / l_new)[:, None] * P_ij @ V_j

        # 更新统计量
        m_i, l_i = m_new, l_new

        # === 写回 HBM ===
        O[i*B_r : (i+1)*B_r] = O_i
        l[i*B_r : (i+1)*B_r] = l_i
        m[i*B_r : (i+1)*B_r] = m_i

return O  # 数学上 ≡ softmax(QK^T / √d) @ V

1.5 KV Cache 原理

在 autoregressive 生成中，每生成一个新 token 都需要重新计算所有前序 token 的 attention。KV Cache 通过缓存已计算的 K 和 V 矩阵，避免重复计算。

为什么只缓存 K 和 V，不缓存 Q？

因为在 autoregressive 生成时，我们只需要计算 新生成 token 的 Q 与所有之前 token 的 K, V 之间的 attention。之前 token 的 Q 不再需要（它们的输出已经确定）。

Attention(qnew, [Kcached; knew], [Vcached; vnew])

内存占用分析

KV Cache 大小 = 2 (K+V) × num_layers × num_heads × head_dim × seq_len × dtype_size

例如：LLaMA-2-70B（80层, 64 heads, head_dim=128, FP16）在 seq_len=4096 时：
KV Cache = 2 × 80 × 64 × 128 × 4096 × 2 bytes = ~10.7 GB per request

代码实现：KV Cache

import torch
import torch.nn as nn

class CachedAttention(nn.Module):
    """带有 KV Cache 的 Attention 实现"""

    def __init__(self, d_model, num_heads):
        super().__init__()
        self.d_model = d_model
        self.num_heads = num_heads
        self.head_dim = d_model // num_heads

        self.W_q = nn.Linear(d_model, d_model)
        self.W_k = nn.Linear(d_model, d_model)
        self.W_v = nn.Linear(d_model, d_model)
        self.W_o = nn.Linear(d_model, d_model)

        # KV Cache: 初始为空
        self.k_cache = None
        self.v_cache = None

    def forward(self, x, use_cache=False):
        """
        x: (batch, seq_len, d_model)
           - Prefill 阶段: seq_len = 完整 prompt 长度
           - Decode 阶段: seq_len = 1 (只有新 token)
        """
        B, L, _ = x.shape

        # 计算当前输入的 Q, K, V
        q = self.W_q(x).view(B, L, self.num_heads, self.head_dim).transpose(1, 2)
        k = self.W_k(x).view(B, L, self.num_heads, self.head_dim).transpose(1, 2)
        v = self.W_v(x).view(B, L, self.num_heads, self.head_dim).transpose(1, 2)

        if use_cache:
            if self.k_cache is not None:
                # Decode 阶段：拼接缓存的 K, V
                k = torch.cat([self.k_cache, k], dim=2)
                v = torch.cat([self.v_cache, v], dim=2)
            # 更新缓存
            self.k_cache = k.detach()
            self.v_cache = v.detach()

        # 标准 attention 计算
        scale = self.head_dim ** -0.5
        scores = torch.matmul(q, k.transpose(-2, -1)) * scale

        # Causal mask (decode 时 q 只有 1 个 token，不需要 mask)
        if L > 1:
            mask = torch.tril(torch.ones(L, k.size(2), device=x.device))
            scores = scores.masked_fill(mask[-L:] == 0, float('-inf'))

        attn = torch.softmax(scores, dim=-1)
        out = torch.matmul(attn, v)

        # reshape 并输出投影
        out = out.transpose(1, 2).contiguous().view(B, L, self.d_model)
        return self.W_o(out)

    def clear_cache(self):
        self.k_cache = None
        self.v_cache = None

model.eval()

    out = model(prompt, use_cache=True)

练习 2：Flash Attention IO 复杂度分析

假设序列长度 N=4096，head 维度 d=128，SRAM 大小 M=100KB（可存储约 M/(2 bytes) = 50K 个 FP16 元素）。

  1. 标准 Attention 的 HBM 读写量是多少？
  2. Flash Attention 的 block size 应如何选择？
  3. Flash Attention 的 HBM 读写量是多少？节省了多少？
提示

标准 Attention 需要：读 Q,K,V (3Nd)，写 S=QKT (N2)，读 S 做 softmax (N2)，写 softmax 结果 (N2)，读 softmax 和 V (N2 + Nd)，写 output (Nd)。

Flash Attention 的 block size Bc ≈ M/(4d)，外循环 N/Bc 次。

完整解答

标准 Attention HBM 访问量：

Step 1: 读 Q[N,d] + K[N,d]: 2Nd = 2×4096×128 = 1,048,576 元素
        写 S[N,N] 到 HBM:   N²  = 4096²       = 16,777,216 元素
Step 2: 读 S[N,N]:           N²  = 16,777,216 元素
        写 P[N,N] 到 HBM:   N²  = 16,777,216 元素
Step 3: 读 P[N,N] + V[N,d]: N²+Nd = 16,777,216 + 524,288 = 17,301,504 元素
        写 O[N,d] 到 HBM:   Nd  = 524,288 元素

汇总:
  读: 2Nd + N² + (N² + Nd) = 3Nd + 2N² ≈ 35,127,808 元素
  写: N² + N² + Nd         = 2N² + Nd   ≈ 34,078,720 元素
  合计: 4N² + 4Nd ≈ 69,206,528 元素 × 2 bytes ≈ 132 MB

注意: V 只在 Step 3 才需要读取, 不是和 Q,K 一起读的!

Flash Attention 配置：

SRAM 容量: M = 50,000 个 FP16 元素
需要存储: K_block (B_c × d) + V_block (B_c × d) + Q_block (B_r × d) + output (B_r × d)
Block size: B_c = M / (4d) = 50000 / (4 × 128) ≈ 97
取 B_c = B_r = 64 (2的幂次，便于对齐)

外循环次数: T_c = N / B_c = 4096 / 64 = 64
内循环次数: T_r = N / B_r = 4096 / 64 = 64

Flash Attention HBM 访问量：

每次外循环加载 K_block + V_block: 2 × B_c × d = 2 × 64 × 128 = 16,384
每次内循环加载 Q_block + O_block: 2 × B_r × d = 16,384
总读写: T_c × (2×B_c×d + T_r × 4×B_r×d)
      = 64 × (16384 + 64 × 32768)
      ≈ O(N^2 × d / B_c) = O(N^2 × d^2 / M)

具体: ≈ N^2 × d / B_c = 4096^2 × 128 / 64 ≈ 33,554,432 元素
相比标准: 33.5M vs 69.7M，节省约 52%

更重要的是: 不需要 O(N^2) 的额外内存存储完整 attention 矩阵！
标准方法需要 4096^2 × 2 bytes = 32MB 额外内存
Flash Attention 只需要 O(N) ≈ 几 KB 额外内存

练习 4（Bonus）：比较 Prefill 和 Decode 阶段的计算特性

解释为什么 Prefill 阶段是 compute-bound，而 Decode 阶段是 memory-bound。提示：考虑每个阶段的 arithmetic intensity（计算量/内存访问量）。

提示

Arithmetic Intensity = FLOPs / Bytes Accessed

Prefill: 输入是完整 prompt (N tokens)，做的是大矩阵乘法

Decode: 输入是 1 个 token，但需要读取整个 KV Cache

完整解答

Prefill 阶段（Compute-bound）：

输入: N 个 token 同时处理 (如 N=4096)
设 D = d_model = 4096, num_heads = 32, h = head_dim = D/num_heads = 128

══ 计算 1: QKV 线性投影 (大 GEMM) ══
  X[N, D] @ W_QKV[D, 3D] → QKV[N, 3D]
  FLOPs = 2 × N × D × 3D = 6ND²
        = 6 × 4096 × 4096² ≈ 412 GFLOPs
  读取 = (X + W) × 2 bytes = (ND + 3D²) × 2 ≈ 128 MB
  AI = 6ND² / [(ND + 3D²) × 2] = 3ND/(N+3D)
     ≈ 3×4096×4096 / (4096+12288) ≈ 3072 FLOPs/byte

══ 计算 2: Attention (per head, 共 32 个 head 并行) ══
  Q_h[N, h] @ K_h^T[h, N] → S[N, N]:   FLOPs = 2N²h per head
  S[N, N] @ V_h[N, h] → O_h[N, h]:     FLOPs = 2N²h per head
  所有 head 总计: num_heads × 4N²h = 4N²D
        = 4 × 4096² × 4096 ≈ 275 GFLOPs
  读取 = (Q + K + V) × 2 bytes = 3ND × 2 ≈ 96 MB  (忽略中间 S, 用 Flash Attention)
  AI = 4N²D / (3ND × 2) = 2N/3 ≈ 2730 FLOPs/byte

══ 汇总 ══
  总 FLOPs ≈ 412 + 275 ≈ 687 GFLOPs
  两部分 AI 都 >> 295 (H100 计算带宽比)
  → Compute-bound ✓  GPU 算力是瓶颈

Decode 阶段（Memory-bound）：

输入: 仅 1 个新 token, 但需读取整个 KV Cache
设已生成 N 个 token, D = 4096, h = 128

主要计算: Attention (q × K_cache^T → scores → × V_cache)
  q: [1, h],  K_cache: [N, h],  V_cache: [N, h]  (单个 head)
  FLOPs = 2 × 1 × h × N + 2 × 1 × N × h = 4Nh
        = 4 × 4096 × 128 ≈ 2M FLOPs (per head)
        × num_heads = 32 → 总计 ≈ 67M FLOPs

  读取 bytes = KV Cache = 2 × N × h × num_heads × 2 bytes
             = 2 × 4096 × 128 × 32 × 2 ≈ 64 MB

Arithmetic Intensity = 67M FLOPs / 64 MB ≈ 1.05 FLOPs/byte

Roofline 判断:
  AI (1.05) << 295 (H100 计算/带宽比) → Memory-bound ✓
  GPU 大部分时间在等 HBM 把 KV Cache 读上来, 计算单元空闲

结论: Prefill AI ≈ 3000, Decode AI ≈ 1
      差了 3000 倍! 这就是为什么需要分开调度。

实际影响：这就是为什么 LLM serving 系统会区分 prefill 和 decode，可能将它们分配到不同的硬件或使用不同的批处理策略（如 continuous batching、chunked prefill）。

import torch
import torch.nn as nn
import torch.nn.functional as F
import math

class MyMultiHeadAttention(nn.Module):
    """从零实现 Multi-Head Attention"""

    def __init__(self, d_model, num_heads):
        super().__init__()
        assert d_model % num_heads == 0
        self.d_model = d_model
        self.num_heads = num_heads
        self.head_dim = d_model // num_heads

        # Q, K, V 线性投影
        self.W_q = nn.Linear(d_model, d_model, bias=True)
        self.W_k = nn.Linear(d_model, d_model, bias=True)
        self.W_v = nn.Linear(d_model, d_model, bias=True)
        # 输出投影
        self.W_o = nn.Linear(d_model, d_model, bias=True)

    def forward(self, x, causal_mask=False):
        B, N, D = x.shape

        # Step 1: 线性投影
        Q = self.W_q(x)  # [B, N, d_model]
        K = self.W_k(x)
        V = self.W_v(x)

        # Step 2: 拆分为多个 head [B, N, d_model] → [B, num_heads, N, head_dim]
        Q = Q.view(B, N, self.num_heads, self.head_dim).transpose(1, 2)
        K = K.view(B, N, self.num_heads, self.head_dim).transpose(1, 2)
        V = V.view(B, N, self.num_heads, self.head_dim).transpose(1, 2)

        # Step 3: Scaled Dot-Product Attention
        scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(self.head_dim)
        # scores shape: [B, num_heads, N, N]

        if causal_mask:
            # 上三角为 True 的 mask
            mask = torch.triu(torch.ones(N, N, device=x.device), diagonal=1).bool()
            scores = scores.masked_fill(mask, float('-inf'))

        attention_weights = F.softmax(scores, dim=-1)
        attention_output = torch.matmul(attention_weights, V)
        # attention_output shape: [B, num_heads, N, head_dim]

        # Step 4: Concat heads → [B, N, d_model]
        output = attention_output.transpose(1, 2).contiguous().view(B, N, D)

        # Step 5: 输出投影
        output = self.W_o(output)

        return output

import torch
import torch.nn as nn
import torch.nn.functional as F
import math
import time

class SimpleTransformerLayer(nn.Module):
    def __init__(self, d_model, num_heads):
        super().__init__()
        self.d_model = d_model
        self.num_heads = num_heads
        self.head_dim = d_model // num_heads
        self.W_q = nn.Linear(d_model, d_model, bias=False)
        self.W_k = nn.Linear(d_model, d_model, bias=False)
        self.W_v = nn.Linear(d_model, d_model, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)

    def forward(self, x, kv_cache=None):
        B, L, _ = x.shape

        # 投影
        q = self.W_q(x).view(B, L, self.num_heads, self.head_dim).transpose(1, 2)
        k = self.W_k(x).view(B, L, self.num_heads, self.head_dim).transpose(1, 2)
        v = self.W_v(x).view(B, L, self.num_heads, self.head_dim).transpose(1, 2)

        # 拼接 KV Cache
        if kv_cache is not None:
            k_cache, v_cache = kv_cache
            k = torch.cat([k_cache, k], dim=2)  # [B, H, past+L, D]
            v = torch.cat([v_cache, v], dim=2)
        new_kv_cache = (k.detach(), v.detach())

        # Attention
        total_len = k.size(2)
        scores = torch.matmul(q, k.transpose(-2, -1)) / math.sqrt(self.head_dim)

        # Causal mask: q 的位置 i 只能看 k 的位置 0..i
        if L > 1:
            # Prefill: 需要完整的 causal mask
            mask = torch.triu(torch.ones(L, total_len, device=x.device), diagonal=total_len - L + 1).bool()
            scores = scores.masked_fill(mask, float('-inf'))
        # Decode (L=1): 新 token 可以看所有过去的 token, 无需 mask

        attn = F.softmax(scores, dim=-1)
        out = torch.matmul(attn, v)

        # 合并 head + 输出投影
        out = out.transpose(1, 2).contiguous().view(B, L, self.d_model)
        out = self.W_o(out)
        return out, new_kv_cache


def generate_with_cache(model, prompt_embeddings, num_new_tokens):
    """使用 KV Cache 的生成"""
    outputs = []

    # Prefill: 处理完整 prompt
    with torch.no_grad():
        out, kv_cache = model(prompt_embeddings, kv_cache=None)
        outputs.append(out)
        # 取最后一个 token 的输出作为下一步输入
        next_input = out[:, -1:, :]  # [1, 1, d_model]

    # Decode: 逐 token 生成
    for _ in range(num_new_tokens - 1):
        with torch.no_grad():
            out, kv_cache = model(next_input, kv_cache=kv_cache)
            outputs.append(out)
            next_input = out  # [1, 1, d_model]

    return outputs


def generate_without_cache(model, prompt_embeddings, num_new_tokens):
    """无 KV Cache 的生成 (每步重算)"""
    outputs = []
    all_embeddings = prompt_embeddings.clone()

    for step in range(num_new_tokens):
        with torch.no_grad():
            # 每次传入完整序列, 不用 cache
            out, _ = model(all_embeddings, kv_cache=None)
            # 取最后一个 token 的输出
            last_out = out[:, -1:, :]
            outputs.append(last_out)
            # 将新输出拼接到序列末尾
            all_embeddings = torch.cat([all_embeddings, last_out], dim=1)

    return outputs

# 1. S = Q @ K^T  →  shape: [batch, n_heads, seq_len, seq_len]
#    这是 attention score matrix, 大小为 O(batch × heads × seq²)
#
# 2. P = softmax(S)  →  shape: [batch, n_heads, seq_len, seq_len]
#    softmax 后的 attention weights, 同样 O(batch × heads × seq²)
#
# 3. dP (backward 时需要保留 P)  →  同上 shape
#

#
# Flash Attention 从不 materialize 完整的 S 或 P 矩阵!
# 它在 SRAM 中以 block 为单位计算:
#   - 每次只加载 Q_block [block_size, head_dim]

# = 4 × 32 × 1,073,741,824 × 4

回答以下关于 KV Cache 工作原理和故障恢复的问题：

  1. 为什么 decode 时不需要重新计算之前所有 token 的 K 和 V？写出 decode step t 的完整计算过程。
  2. 如果因为 preemption 丢失了 KV cache，re-prefill 的计算量是多少？（以 FLOPs 表示，给出 70B 模型、4096 tokens 的具体数值）
  3. 设计一个 "partial re-prefill" 策略：只恢复最近 W 个 token 的 KV，利用 sliding window attention 的近似。分析其 trade-off。
完整解答 
# KV Cache 的 "增量" 本质

## Part 1: Decode Step t 的完整计算
#
# 核心观察: K = X @ W_K, V = X @ W_V 是逐 token 独立的线性变换
# token i 的 K_i = x_i @ W_K 只依赖 x_i, 与其他 token 无关
# 因此, 之前 token 的 K, V 值永远不会改变, 可以安全缓存
#
# Decode step t 的计算:
# ─────────────────────────────────────────────
# 输入: x_t (当前新 token 的 embedding)
# 已有: K_cache = [K_0, K_1, ..., K_{t-1}]  shape: [t, head_dim]
#        V_cache = [V_0, V_1, ..., V_{t-1}]  shape: [t, head_dim]
#
# Step 1: 计算新 token 的 Q, K, V (只需要 1 个 token!)
#   q_t = x_t @ W_Q    # shape: [1, head_dim]
#   k_t = x_t @ W_K    # shape: [1, head_dim]
#   v_t = x_t @ W_V    # shape: [1, head_dim]
#
# Step 2: 更新 cache
#   K_cache.append(k_t)  → shape: [t+1, head_dim]
#   V_cache.append(v_t)  → shape: [t+1, head_dim]
#
# Step 3: 计算 attention (q_t attend to ALL cached keys)
#   scores = q_t @ K_cache.T  # [1, head_dim] × [head_dim, t+1] → [1, t+1]
#   scores = scores / sqrt(head_dim)
#   weights = softmax(scores)  # [1, t+1]
#
# Step 4: 加权求和
#   output = weights @ V_cache  # [1, t+1] × [t+1, head_dim] → [1, head_dim]
#
# 关键: 不需要重新计算 K_0..K_{t-1}, 因为它们已经在 cache 中!

## Part 2: Re-prefill 计算量
#
# Prefill N tokens 的 FLOPs ≈ 2 × N × model_params
# (每个参数对每个 token 做一次乘法和一次加法)
#
# 70B model, N=4096 tokens:
N = 4096
params = 70e9
re_prefill_flops = 2 * N * params
# = 2 × 4096 × 70×10^9
# = 573,703,577,600 ≈ 574 TFLOP

# H100 峰值 FP16 算力: ~990 TFLOPS (with sparsity), 实际利用率 ~60%
# 有效算力 ≈ 600 TFLOPS
re_prefill_time = re_prefill_flops / 600e12
# = 574e12 / 600e12 ≈ 0.96 秒
#
# 这意味着: 丢失 KV cache 后, 需要花 ~1 秒重新 prefill
# 对实时服务来说这是非常大的延迟!

## Part 3: Partial Re-prefill with Sliding Window
#
# 策略: 只恢复最近 W 个 token 的 KV cache
# ─────────────────────────────────────────────
# 假设原始 context 有 N=4096 tokens, 选择 W=1024
#
# Re-prefill 计算量: 2 × W × params = 2 × 1024 × 70e9 ≈ 143 TFLOP
# 时间: 143e12 / 600e12 ≈ 0.24 秒 (节省 75%)
#
# 工作原理:
#   - 将 token[0..N-W-1] 的 KV 视为丢失 (attention weight 设为 0)
#   - 只对 token[N-W..N-1] 做 prefill, 恢复局部 KV cache
#   - 后续 decode 时, 新 token 只能 attend 到最近 W 个位置
#
# 适用场景:
#   - Mistral 等使用 sliding window attention 的模型 (天然只看最近 W 个)
#   - 对话场景中, 最近的 context 通常最重要
#
# Trade-offs:
#   ✓ 恢复时间从 O(N) 降到 O(W)
#   ✓ 恢复后的 KV cache 内存也更小
#   ✗ 丢失长距离依赖 (如果模型依赖早期 token 的信息)

本章小结

  • Self-Attention 通过 Q, K, V 矩阵乘法实现 token 间的信息交互
  • Multi-Head Attention 将注意力拆分到多个子空间，增强表达能力
  • Flash Attention 通过 tiling + 在线 softmax 减少 HBM 访问，是 IO-aware 的精确算法
  • KV Cache 通过缓存已计算的 K, V 避免 decode 阶段的重复计算
  • Chat Templates 定义了多轮对话的 token 化格式，对模型输出质量至关重要
